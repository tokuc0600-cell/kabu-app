# Note Workflow Agent

## 役割
Leonのnote記事の生成・管理を担うエージェント（ヘッダー画像生成は対象外。下記参照）。
スプレッドシートに収集した海外IR・ニュース情報をもとにした時事ネタ記事と、
kabu_appの開発体験をベースにした開発トレース記事の2系統を扱う。

---

## ディレクトリ構成

```
note_workflow/
├── CLAUDE.md                  # この指示ファイル
├── fetch_gdoc.py              # Google Sheets/DocsをDrive API経由で取得するスクリプト
├── prompts/
│   ├── note_agent.md          # 記事生成プロンプト：開発トレース記事
│   ├── news_agent.md          # 記事生成プロンプト：時事ネタ記事
│   └── x_post_agent.md        # X投稿生成プロンプト（速報・前振り用）
├── drafts/                    # note下書き（Claude.aiからの出力を配置する場所）
│   └── [slug].md
├── published/                 # note公開済みアーカイブ
│   └── [slug].md
├── x_drafts/                  # X投稿ドラフト（投稿待ち）
│   └── YYYY-MM-DD-slug.md     # 1ファイル = 1テーマ分の投稿案セット
├── x_posted/                  # X投稿済みアーカイブ
│   └── YYYY-MM-DD-slug.md
├── dev_logs/                  # 開発トレース記事に書き起こす前の中間素材（生のやり取りメモ）
│   └── YYYY-MM-DD-トピックslug.md
└── assets/
    ├── headers/               # ヘッダー画像（2026-06-25〜：Claude Codeでは生成しない。別の方法でレオンさんが作成）
    │   └── [slug].png
    └── images/                # 記事本文用の画像ファイル
        └── [slug]/            #   記事スラッグごとにディレクトリを切る
```

## ファイル命名規則

| 種別 | 命名パターン | 例 |
|------|------------|-----|
| 時事ネタ記事 | `news-[内容].md` | `news-accenture-q3-earnings.md` |
| 開発トレース記事 | `dev-[内容].md` | `dev-ema-backtest-phase-b.md` |
| 公開済み | 同上スラッグ | — |
| ヘッダー画像 | `[slug].png` | `news-accenture-q3-earnings.png` |
| X投稿ドラフト | `YYYY-MM-DD-[内容slug].md` | `2026-06-28-nvidia-earnings.md` |

- スラッグはハイフン繋ぎの英語（日付は含めない。公開日はgitのコミット履歴で追う）
- プレフィックス `news-` / `dev-` で記事タイプを識別する
- 既存記事の改稿は同じスラッグのファイルを直接編集（gitが履歴を保持）
- X投稿ドラフトは日付をプレフィックスにする（1日複数ファイルを区別するため）

## 記事生成フロー（全体像）

### 時事ネタ記事（news_agent.md使用）

**ネタ収集ソースは2系統（2026-06移行）**：
- スプレッドシート：Manus（ニュース自動収集AIエージェント）の収集分。継続運用。
- Supabase：その他（GAS等）の収集分。スプレッドシートから一部移行。

```
① ニュース自動収集
   └ Manus → スプレッドシートに書き込み
   └ その他（GAS等） → Supabaseに書き込み
      ↓
② Claude.ai（チャット）
   └ スプレッドシート：Drive MCP経由で読み込む（Manus収集分）
   └ Supabase：Supabase MCP経由で読み込む（その他収集分）
   └ 両方のソースを合わせてテーマ候補を提示する（候補ごとに出典＝Manus/Supabaseを明記）
   └ レオンさんが最終決定
   └ web検索で最新情報を補完
   └ news_agent.mdのルールに従って記事ドラフトを生成
   └ drafts/news-[slug].md としてMarkdownファイルを出力（ダウンロード）
      ↓
③ レオンさん（手作業・1ステップのみ）
   └ ダウンロードした.mdファイルをリポジトリの drafts/ に配置
      ↓
④ Claude Code
   └ 「drafts/news-[slug].mdをnews_agent.mdのルールで仕上げて」と指示
   └ 文体統一・画像挿入プレースホルダー・表PNG化・誤字チェック
   └ assets/images/news-[slug]/ を用意（ヘッダー画像は生成しない。別の方法でレオンさんが作成）
   └ git add → commit
```

※Supabase MCPサーバーは設定済み（2026-06-23、`claude mcp add` で `--scope user` 登録、read-only）。②のSupabase参照ステップは運用中。

#### 記事案リストの自動生成（generate_proposals.py）

②のテーマ候補提示を補助するため、Supabase（japan_ir / overseas_ir）から直近ニュースをテーマ
（AI/投資/会計/金融/FX）でスコアリングして候補リストを出力するスクリプトを用意している。

```bash
uv run python note_workflow/generate_proposals.py
# → note_workflow/drafts/proposals_YYYYMMDD.md を生成
```

## note記事案レビュー（Web Claude用）

`proposals_YYYYMMDD.md` をWeb Claudeにアップロードして以下を貼る：

> このproposalsファイルの各記事案にアイデアを補完して。
> テーマはAI・投資・会計・金融・FX。
> ノイズ（Form 8.x、小粒バイオ決算、生活系ニュース）は除外してよい。
> 読者は投資初心者〜中級者の日本人。
> 文体は必ず「です・ます調」で統一して（だ・である調は使わない）。一人称は「僕」、二人称は使わない。
> 初心者を読者として想定してよいが、教える目線（「〜しよう」「〜するべき」等）にはせず、
> 「〜という考え方もある」のように判断は読者に委ねる書き方にして。
> 投資関連で「知っておくべき」という言い回しは使わない。
> 以上を考慮して再構成して。

## X投稿案生成（Web Claude用）

**noteと同じ `proposals_YYYYMMDD.md` をソースに使う**。note記事案レビューと並行して実施できる。

`proposals_YYYYMMDD.md` をWeb Claudeにアップロードして以下を貼る：

> このproposalsファイルを使って、X（旧Twitter）投稿案を生成して。
> x_post_agent.md の指示に従って生成すること（別途共有するか、以下のルールを適用して）。
>
> ルール：
> - 1投稿 140字以内
> - 一人称は「僕」、断定を避ける、投資アドバイスは書かない
> - 各ニュースにつき「速報型」「問いかけ型」を各1つ、必要に応じて「スレッド展開型」も生成
> - note連動型はURLのプレースホルダー（[note URL をここに追記]）を入れる
> - ノイズ（Form 8.x、小粒バイオ決算）は除外
> - 出力ファイル名は x_posts_YYYYMMDD.md 形式で

生成された `x_posts_YYYYMMDD.md` をダウンロードして `note_workflow/x_drafts/YYYY-MM-DD-slug.md` に配置する。

### 開発トレース記事（note_agent.md使用）

```
① kabu_appの開発が一区切りついたタイミング（動いた・検証できた・結果が出た）
      ↓
② Claude.ai（チャット）
   └ 開発内容・背景をレオンさんから口頭で共有
   └ note_agent.mdのルールに従って記事ドラフトを生成
   └ drafts/dev-[slug].md としてMarkdownファイルを出力（ダウンロード）
      ↓
③ レオンさん（手作業・1ステップのみ）
   └ ダウンロードした.mdファイルをリポジトリの drafts/ に配置
      ↓
④ Claude Code（時事ネタ記事と同じ仕上げフロー）
```

**Geminiはこのフローでは使用しない。**
Claude.aiがテーマ選定・ドラフト生成を担い、Claude Codeが仕上げ・Git管理を担う2段構成。

## 記事生成時の自動処理手順（Claude Code側）

Claude Codeは `drafts/[slug].md` が配置されたら、以下をこの順番で実行する。
**トリガーは初回の仕上げ依頼だけでなく、その後の記事編集依頼（タイトル修正・文体修正など）でも同様**。
編集依頼を受けた時点で `assets/images/[slug]/` が存在しなければ、その場で作成する。

```
1. drafts/[slug].md を読み込む
2. スラッグのプレフィックスで使用するプロンプトを判断する
   - news- → prompts/news_agent.md を読み込む
   - dev-  → prompts/note_agent.md を読み込む
3. 文体をです・ます調に統一する
4. 記事構成を分析し、効果的な箇所に（画像挿入：[意図]）プレースホルダーを挿入する
5. 表が必要な箇所は（画像挿入：[表タイトル] PNG画像化）に置き換える
6. 誤字脱字・句読点をチェックする
7. drafts/[slug].md を上書き保存する
8. assets/images/[slug]/ ディレクトリを用意する（画像作成はレオンさんが手動で行う）
   ※ヘッダー画像（assets/headers/）はClaude Codeでは生成しない（2026-06-25〜、別の方法でレオンさんが作成）
9. git add drafts/ assets/images/
10. git commit -m "add: draft [slug]"
```

## 執筆ルール

### 文体
- 文末は**です・ます調**で統一する
- 一人称は「僕」、二人称は使わない
- 話し言葉に近い、平易な日本語を使う
- 専門用語は初出時にカッコ書きで補足する
- 市況・投資判断は断定を避け、「〜かもしれません」「〜な気がします」などの控えめな表現を使う
- リストラ・人員削減を扱う場合、企業側の軽い言い回しを避け、影響を受ける人の立場が伝わる表現を選ぶ
- 投資アドバイスは「ポジションなし」「ポジションあり」の2パターンを必ず書き分ける
- 現象の分かりにくさ・難易度を読者の「初心者かどうか」に結びつける表現（「初心者が迷いやすい」等）は使わない。専門用語の補足など説明の平易さは初心者を意識してよいが、現象自体の難しさは経験に関係ないことが多い
- 投資関連の文脈で「（〜が）知っておくべき」という言い回しは使わない。「べき」は内容が正しい・確実であることを保証するような響きを持つが、投資の見立てはそうとは限らないため、誤った安心感を与えるリスクがある。タイトル・見出しでも同様
- 初心者を読者として想定してよいが、教える目線（「〜しよう」「〜するべき対策」のように読者に指示する言い回し）にはしない。「〜という考え方もある」「〜も一つの選択肢になる」のように、判断は読者に委ねる書き方にする
- すでに広く知られている一般的な論点を整理する場合は、「ご存知の方も多いかもしれませんが」のように、読者が既に知っている前提を踏まえた上で改めて整理する、というニュアンスを入れる。読者がまだ知らない前提で説明する書き方は避ける
- 「〜という習慣をつけると見え方が変わってきます」のように、読者がその習慣を持っていない前提で教え諭すニュアンスの結び方は避ける。リスクや視点は「常に意識しておくべきもの」として書く
- 自分の活動（ツール開発など）に言及する場合、「別の記事で」のような他記事の存在を前提にした言及はしない。その記事だけを読む読者でも文脈が分かるように、自己完結した説明にする
- 「（努力して）両方を磨いていきたい」のような、努力すれば把握・予測できるという含みのある表現は避ける。読み切れない・予測できないリスクは常に存在する前提で書く
- 業界の構造変化（AI脅威など不確実性の高いテーマ）に触れる場合は、断定せずアナリスト等の一般的な見解を調べた上で、賛否両論を踏まえて書く
- 海外レポートを読み解く記事で「〜を日本語でまとめます」のような表現は使わない。記事自体がそもそも日本語で書かれているため不要な言い回しであり、誤訳を疑わせる印象も与える。代わりに「整理してお伝えします」「読み解いていきます」など、要約・分析の行為そのものを指す表現にする
- 英語圏の記事・レポート由来の比喩や言い回し（例：「市場の配管（plumbing）」「ベアマーケットのように感じられる」）は直訳しない。日本語の感覚で意味が伝わる表現に言い換える
- 「需給イベント」のような専門用語は、カッコ書きの一言補足だけで済ませず、仕組み自体（誰が・なぜ・どういう理由で動くのか）を平易な言葉で噛み砕いて説明する
- 「こういう視点こそ海外の一次情報を読む価値があります」のように、特定の情報源・手法（海外レポート、英語記事、AI要約など）の価値を読者に売り込むような結び方は避ける。気づき・感想にとどめ、そこから何を読み取るかは読者の判断に委ねる
- 自分が実際に行っている投資手法（信用取引など）に関連するテーマに触れる場合は、当事者目線でメリット・デメリットや、現物投資との視点の違いまで掘り込む
- 記事のまとめ的な見出し（「〜学んだこと」等）は学習者目線になりすぎる表現を避け、「〜感じたこと」など実体験に基づく所感として伝わる表現にする
- 政府要人・企業経営者など公的な役職者の発言を扱う場合、個人名ではなく役職名（「財務相」等）で表記する。役職者は交代するため、個人名を入れると記事が古く・不正確に見えるリスクがある
- 「今週」「今月」「最近」のように、読まれるタイミングに依存する相対的な時間表現は使わない。記事はいつ読まれるか分からないため、絶対的な日付（「2026年6月22日」等）で表記する
- 具体的な数値や裏付けを示せない内容は、ぼんやりした記述のまま残さず、書かないか削除する（「評価額が増える方向に影響します」のような曖昧な一般論は要素として弱い）

### 画像挿入の扱い
- 視覚的な区切り・理解補助として効果的な箇所に `（画像挿入：[挿入意図の説明]）` を挿入する
- 画像そのものの生成・選定は行わない（レオンさんが手動で作成）
- プレースホルダーの説明は、本文の数値・項目名を明記した具体的な内容にする（「主要指標比較表」のような曖昧な表現は不可。どの行・どの数値を比較するのかまで書く）
- プレースホルダー内で要素数を述べる場合（「3つの要因」等）は、対応する本文の項目数と必ず一致させる
- プレースホルダーは、記事ファイル全体を読み込まなくてもその1箇所だけで画像生成が完結するように書く（画像生成AIに記事全文を読ませると意図通りに生成されないため）。1つのプレースホルダーの中に、以下を全て明記する：
  - 画像タイトル（何の比較・図解か）
  - 行数・要素数の構成（例：「3行構成」）
  - 各行・各要素の具体的なラベルと数値（本文中の数値をそのまま転記する）
  - 表現形式（横棒グラフ／表／対比図解など）
  - 例：`（画像挿入：「SpaceXのNasdaq-100組み入れに伴う強制的な買い予想額」比較表 PNG画像化 — 3行構成。1行目「BNP Paribas：初月の受動的買い＝80億ドル」、2行目「BNP Paribas：関連指数イベント全体＝300億ドル」、3行目「SpotGamma：指数ファンドの強制的な買い＝220億〜270億ドル」。横棒グラフ形式で金額の大小が一目で比較できるようにする）`
- 確率・想定レンジなど将来予測を含む比較表・図解には、あくまで想定であり結果を保証しない旨の注記キャプションをプレースホルダーの説明内に含める
- **毎回必ず実行する固定ルール**

### 表の扱い
- note本文にMarkdownの表を直接埋め込まない
- `（画像挿入：[表タイトル] PNG画像化）` に置き換える
- **毎回必ず実行する固定ルール**

### 画像ファイルの保存運用
- 完成した画像ファイルは `assets/images/[slug]/` に保存する
- Git管理下に置き、履歴として残す
- noteへのアップロードはnote編集画面上で手動で行う

### 下書き素材（Google Drive / Supabase）の位置づけ
- スプレッドシートはDrive API/Drive MCP経由で**読み取り専用**で参照する（Manus収集分）
- Supabaseは**Supabase MCP経由・読み取り専用**で参照する（その他収集分）
- Google Drive側・Supabase側いずれもファイル・レコードへの書き込みは行わない（自動収集はManus/GAS側の責務）

### 誤字脱字・句読点チェック
- drafts/ への保存前に最終チェックを行う

### Xポスト案
- **note記事末尾へのXポスト案の埋め込みはしない**（2026-06-24廃止・方針変更）
- X投稿は note とは独立したフロー（`x_drafts/`）で管理する（2026-06-28〜）
- 役割分担：**X投稿 = 前振り・速報、note = じっくり考察**
  - X投稿：ニュースが出たら即日投稿が原則。タイムリーさを最優先
  - note記事：タイムリーさより深さ・考察を優先。推敲に数日かけてよい
  - note公開後、対応するX投稿の「note連動型」投稿案にURLを追記して投稿する
- 既存の note 下書き内に残っているXポスト案の節は、次回編集時に削除する
- X投稿ドラフトの生成方法：`x_post_agent.md` を参照

### 免責事項の固定文言
- 記事末尾の免責事項は、必ず以下の文言をそのまま使う（改変しない）
  ```
  ※この記事の内容は個人の見解に基づくものです。特定の銘柄の売買を推奨するものではありません。投資に関する最終的な判断はご自身でお願いします。
  ```

### 禁止事項
- Google Drive側への書き込み・編集は行わない
- Supabase側への書き込み・編集は行わない
- note本文中にMarkdownの表をそのまま残さない
- 画像ファイルの生成・選定はこのフローで行わない

## 記事タイプ別の運用方針

### 時事ネタ記事（news-プレフィックス）
- ニュースが出たら鮮度優先で出す（1週間以上経ったネタは価値が落ちる）
- 記事末尾に毎回「kabu_app開発記事への誘導」を入れる（読者を開発トレース記事へつなぐ）
- 出し切り。後から加筆・改稿はしない

### 開発トレース記事（dev-プレフィックス）
- 反映タイミング：①動いた（日誌型）②検証できた（マイルストーン型）③結果が出た（総括型）
- 日誌型は短くてよい。失敗談・試行錯誤もネタになる
- 日誌型の積み上げが集大成記事のたたき台になる

#### 開発メモ（dev_logs）— 記事に書き起こす前の素材保存

開発トレース記事は1回のセッションで完結せず、機能追加・修正のたびに節を追記していく運用（2026-06-25〜）。
セッション中の生のやり取り（IDEの画面をそのままコピー&ペーストしたもの等）を直接記事に転用すると、
Markdownのエスケープが崩れたり、ツールの内部ログと実際の指示文が混在したりして使いにくいため、
**記事に書き起こす前の中間素材**として `note_workflow/dev_logs/` に整理したメモを残す。

```
1. kabu_appの開発が一区切りついたタイミング（動いた・検証できた・トラブルが解決した、等）
      ↓
2. Claude Codeに「今回のセッションのメモをdev_logsに残して」と伝える
   └ note_workflow/dev_logs/YYYY-MM-DD-トピックslug.md を以下のテンプレートで作成
      ・概要（1-2行）
      ・実際に出した指示（生の文面。整形しない）
      ・起きたこと・気づいたこと
      ・原因
      ・直し方
      ・（あれば）トラブルの詳細
      ↓
3. 記事を書く/追記するタイミングで、Claude Codeに該当メモを読み込ませて
   pf-backtest-engine.md（または新規dev-記事）の節として書き起こす
      ↓
4. 反映したメモのファイル末尾に「→ [記事slug]の「節タイトル」に反映済み」を追記する
   （同じメモを二重に書き起こさないようにするため）
```

- 生ログのコピー&ペーストではなく、その場でClaude Codeに整理させて書き出す（Markdownが崩れない・後で記事化しやすい形になる）
- メモ自体は「です・ます調」の執筆ルールに従わなくてよい（読み物ではなく素材のため）

## Leonがやること（手作業・最小化済み）

### note記事フロー
1. `uv run python note_workflow/generate_proposals.py` を実行する
2. 生成された `drafts/proposals_YYYYMMDD.md` をWeb Claudeにアップロードし、記事案レビューの指示を貼る
3. テーマを最終決定してClaude.aiに伝える
4. Claude.aiが出力した `.md` ファイルをダウンロードして `drafts/` に配置する
5. Claude Codeに「drafts/[slug].mdを仕上げて」と伝える
6. `assets/images/[slug]/` に本文用画像を作成・保存する
7. noteに記事をコピペし、ヘッダー画像・本文画像をアップロードする
8. noteで公開ボタンを押す
9. 「publishedに移動して」とClaude Codeに伝える

### X投稿フロー（noteと並行・独立）
1. 同じ `proposals_YYYYMMDD.md` をWeb ClaudeにアップロードしX投稿案の指示を貼る
2. Web Claudeが出力した `x_posts_YYYYMMDD.md` をダウンロードして `x_drafts/YYYY-MM-DD-slug.md` に配置する
3. Claude Codeに「x-draft [slug] をコミットして」と伝える
4. `x_drafts/` を開いて投稿案を確認し、Xに投稿する（即日が原則）
5. note公開後、対応する「note連動型」投稿案にURLを追記してXに投稿する
6. 「posted: x [slug]」とClaude Codeに伝える（x_posted/に移動してコミット）
