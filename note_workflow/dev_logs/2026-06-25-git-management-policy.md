# 2026-06-25 dev_logsをgit管理に含めるかの判断指針

## 概要
dev_logsメモ作成後、「git管理に含めるかどうかの判断基準は何か」という質問を受け、このプロジェクトの`.gitignore`・CLAUDE.md・note_workflow/CLAUDE.mdの既存ルールから判断指針を整理した。

## 実際に出した指示
```
git管理に含めるか同課の判断の指針を教えて
```
（「同課」はおそらく「同課（どうか）」の誤字で「含めるか含めないか」の意図）

## 起きたこと・気づいたこと
- プロジェクトには明文化された「何をgit管理に含めるか」の一覧ルールは無かったが、`.gitignore`の内容とnote_workflow/CLAUDE.mdの運用フロー記述から、暗黙の判断軸を抽出できた。
- `dev_logs/`はnote_workflow/CLAUDE.mdの運用ルール上、メモ末尾に`→ [記事slug]の「節タイトル」に反映済み`という追跡用の一文を書く設計になっており、これはコミット履歴として残ることを前提にしている（反映済みかどうかを後から辿れる必要があるため）。

## 原因
（このタスクは「バグの原因」ではなく方針整理のため、該当なし）

## 直し方
以下の判断指針を整理して回答した：

**含めない（.gitignore対象）**
1. 機密情報（`credentials/`、`*.json`系、`secrets.toml`）
2. 再生成可能なキャッシュ・出力物（`backtest/data/`、`investment-report/input,output/`）
3. 環境・ツール依存ファイル（`.venv/`、IDE設定、OSファイル）
4. 一時ログ（`*.log`、`*.tmp`）

**含める**
1. 動作を定義するもの（ソースコード・設定・CLAUDE.md等のドキュメント）
2. 後で参照する価値がある中間素材（`note_workflow/drafts/`・`published/`・`dev_logs/`）— 「公開日はgitのコミット履歴で追う」という方針のため
3. 記事用に手作りした画像（`assets/images/`、`assets/headers/`）

この指針に基づき、今回作成した2件のdev_logsメモ（`2026-06-25-yfinance-rate-limit-fix.md`・`2026-06-25-stock-short-position-support.md`）はコミット対象と判断し、実際にコミットした。
